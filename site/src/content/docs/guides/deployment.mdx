## Overview

The Waza Platform can be deployed to Azure using the **Azure Developer CLI** (`azd`). This guide covers prerequisites, deployment steps, and post-deployment configuration.

## Prerequisites

Before deploying, ensure you have:

1. **Azure Subscription** — An active Azure subscription with sufficient quota for Container Apps, Cosmos DB, and Storage
2. **Azure Developer CLI** — Install `azd` from [microsoft.github.io/azure-dev](https://microsoft.github.io/azure-dev)
3. **GitHub Application** — Register a Waza GitHub App for OAuth login:
   - Go to **GitHub → Settings → Developer Settings → OAuth Apps**
   - Click **New OAuth App**
   - Set Authorization Callback URL to `https://{your-domain}/auth/callback`
   - Save the **Client ID** and **Client Secret** — you'll need these during deployment
4. **Git** — To clone the repository
5. **Appropriate Azure Permissions** — Owner or Contributor role on the subscription

## Deployment Steps

### 1. Clone and Initialize

```bash
git clone https://github.com/microsoft/waza.git
cd waza
azd init
```

`azd init` will prompt you for:
- **Environment name** (e.g., `prod`, `staging`) — used for resource naming
- **Azure subscription** — select your target subscription
- **Azure region** — choose a region (e.g., `eastus`, `westus2`)

### 2. Set Environment Variables

Create a `.env` file in the project root with your GitHub OAuth credentials and other secrets:

```bash
GITHUB_CLIENT_ID=your_github_client_id
GITHUB_CLIENT_SECRET=your_github_client_secret
GITHUB_APP_ID=your_github_app_id
```

Additionally, set these via `azd env set`:

```bash
azd env set GITHUB_CLIENT_ID "your_github_client_id"
azd env set GITHUB_CLIENT_SECRET "your_github_client_secret"
azd env set GITHUB_APP_ID "your_github_app_id"
```

### 3. Provision and Deploy

Run the deployment:

```bash
azd up
```

This command will:
1. **Provision** Azure resources (Container Apps, Cosmos DB, Storage Account, Key Vault)
2. **Build** the Go backend and React frontend
3. **Deploy** containers to Azure Container Apps
4. **Display** the platform URL once complete

Deployment typically takes 5–10 minutes.

### 4. Verify Deployment

After `azd up` completes, test the platform:

```bash
# Get the platform URL
azd env list

# Test the health endpoint
curl https://{your-platform-url}/health
```

You should see a `200 OK` response with health status.

### 5. Configure Post-Deployment

#### Update GitHub OAuth Callback URL

If your platform URL changed after deployment, update your GitHub OAuth App:

1. Go to **GitHub → Settings → Developer Settings → OAuth Apps**
2. Edit the Waza OAuth App
3. Update **Authorization Callback URL** to `https://{your-deployed-domain}/auth/callback`

#### Enable Storage Connection

Users will need to configure Azure Storage connections in the platform UI. If using a centralized storage account:

1. Create an Azure Storage Account (if not already created by `azd up`)
2. Go to platform **Settings → Connections**
3. Add your storage account name and container name
4. Click **Test Connection**

#### Configure ADC Sandbox Access

For remote execution on ADC sandboxes:

1. Obtain your ADC API key from your ADC environment
2. Go to platform **Settings → Execution**
3. Enter your ADC API key and sandbox endpoint
4. Click **Test Connection**

## Monitoring and Logs

### View Application Logs

```bash
azd monitor
```

This opens Application Insights with logs from the Go backend and React frontend.

### Check Resource Health

View deployed resources:

```bash
azd resource show
```

Monitor specific services:

```bash
az container app logs show \
  --name waza \
  --resource-group <resource-group-name>
```

### Common Issues

| Issue | Solution |
|-------|----------|
| GitHub OAuth callback fails | Ensure callback URL matches GitHub OAuth App settings |
| Storage connection fails | Verify storage account name and ensure your identity has access |
| ADC execution timeout | Check ADC API key and sandbox endpoint configuration |
| High container app costs | Check for long-running eval tasks; consider adjusting worker count or timeout limits |

## Custom Domain Setup

To use a custom domain instead of the default Azure Container Apps domain:

1. **Add custom domain to Container App:**
   ```bash
   az containerapp hostname bind \
     --name waza \
     --resource-group <resource-group-name> \
     --hostname your-domain.com
   ```

2. **Update DNS records:**
   - Point your domain's CNAME or A record to the Container App's FQDN
   - DNS propagation typically takes 15–30 minutes

3. **Update GitHub OAuth App:**
   - Set callback URL to `https://your-domain.com/auth/callback`

## Scaling

The platform auto-scales based on demand:

- **Min replicas:** 2 (for high availability)
- **Max replicas:** 10 (configurable in `infra/main.bicep`)

To adjust scaling limits:

```bash
az containerapp update \
  --name waza \
  --resource-group <resource-group-name> \
  --min-replicas 3 \
  --max-replicas 20
```

## Cleanup

To remove all Azure resources and avoid ongoing costs:

```bash
azd down
```

This deletes the resource group and all contained resources. **This action is irreversible** — ensure you have backed up any important data.

## Troubleshooting

### Deployment Fails with "Insufficient Quota"

If you hit Azure quota limits:

1. Check your subscription's current usage in the Azure Portal
2. Request a quota increase for Container Apps, Cosmos DB, or Storage
3. Retry `azd up`

### GitHub OAuth Not Working

1. Verify **Client ID** and **Client Secret** are correct
2. Confirm callback URL matches your deployment domain
3. Check Application Insights logs for OAuth errors

### Storage Connection Fails

1. Verify storage account name is correct
2. Ensure your identity has **Storage Blob Data Contributor** role
3. Check that the container exists (create it if needed)

For additional help, see the [Waza Platform guide](./platform.mdx) or open an issue on [GitHub](https://github.com/microsoft/waza).
